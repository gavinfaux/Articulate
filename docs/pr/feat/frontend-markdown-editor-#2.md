# **PRD: Modernization of Mobile Articulate Markdown Editor **

* **Document Owner:** You
* **Date:** June 30, 2025
* **Status:** Approved for Implementation

## 1. Introduction & Problem Statement

The current Markdown Editor component within the Articulate package is built on the end-of-life AngularJS 1.2 framework and the deprecated Material Design Lite (MDL) library. This presents challenges for maintainability, security, and user experience. This project will replace the legacy component with a modern, secure, and maintainable solution that prioritizes simplicity and performance by avoiding a complex frontend build process.

## 2. Goals & Objectives

* **Decommission Legacy Code:** Completely remove all AngularJS 1.2 and MDL code.
* **Implement a "No Build" Frontend:** The new frontend will be built using **Petite-Vue** for reactivity directly within the `.cshtml` view. This eliminates the need for Node.js, Vite, or any frontend build/compile steps.
* **Adopt a Lightweight CSS Framework:** The UI will be styled with **Pico.css** for a clean, modern, class-less, and responsive design with minimal overhead.
* **Enhance Security & API Design:** The backend will use two separate, modern ASP.NET Core API controllers. One for authentication and one for data operations, with a strongly-typed client generated by **`@hey-api/openapi-ts`**.
* **Achieve Functional Parity:** The new component will replicate all features: login, Markdown editing, image/camera uploads, metadata entry, and publishing.

## 3. Scope

**In-Scope:**

* A complete rewrite of the frontend component within a single `.cshtml` view using Petite-Vue and Pico.css.
* Replacement of legacy API endpoints with two new, secure API controllers (`ControllerBase` for auth, `ManagementApiControllerBase` for data).
* Generation of a typed API client using `@hey-api/openapi-ts`.
* Integration of `EasyMDE` for a rich Markdown editing experience.

**Out-of-Scope:**

* Adding new features not present in the original component.
* Changes to the core Articulate C# services or data structures.

## 4. Technical Architecture & Implementation Plan

The architecture consists of two backend controllers and a single `.cshtml` file that contains all the frontend HTML, CSS, and JavaScript logic.

### 4.1. Backend Implementation

The backend architecture is final and consists of two separate controllers. **(This section is identical to our previous correct plan).**

**A. Authentication Controller (`MarkdownEditorAuthController.cs`)**

* **Inherits:** `Microsoft.AspNetCore.Mvc.ControllerBase`
* **Purpose:** Handles user login/logout and session status.
* **Authorization:** The `Login` method is public; `GetStatus` and `Logout` are protected by the `BackOfficeAccess` policy.

**B. Data Controller (`BlogPostApiController.cs`)**

* **Inherits:** `Umbraco.Cms.Api.Management.Controllers.ManagementApiControllerBase`
* **Purpose:** Handles the creation of content and media.
* **Authorization:** Methods are protected by specific policies like `ContentPermissionByResource` to ensure the user has the required permissions for the action.

**(NOTE: The full C# code for these two controllers from our previous discussion is correct and should be used as-is).**

### 4.2. Frontend Implementation

This is the core of the new plan. The entire frontend will be self-contained in the `.cshtml` view. There is **no frontend build step**.

* **A. API Client Generation (One-time Manual Step)**

Before writing the frontend, the developer will generate the typed API client.

1. Run the Umbraco application locally.
2. Install the `openapi-ts` tool globally or via `npx`: `npm install -g @hey-api/openapi-ts`
3. Run the command, pointing to the local Swagger URL:

    ```bash
    openapi-ts --input http://localhost:PORT/swagger/v1/swagger.json --output ./App_Plugins/Articulate/Assets/api-client.js --client fetch
    ```

4. This creates a single `api-client.js` file that we can reference.

## **PRD Addendum: Implementation Details**

This addendum provides specific guidance for the two most complex components of the project: the `ManagementApiControllerBase` for data operations and the `EasyMDE` integration.

### 1. Detailed Guidance for `ManagementApiControllerBase`

While `ControllerBase` is perfect for the simple auth controller, using `ManagementApiControllerBase` for the data controller provides significant advantages that a developer needs to understand and leverage.

**A. Core Benefits to Leverage:**

1. **Standardized Error Handling (Problem Details):** This is the most important benefit. Instead of returning simple `400 Bad Request` or `500 Internal Server Error` responses, this base class automatically returns a standardized [RFC 7807 Problem Details](https://tools.ietf.org/html/rfc7807) JSON object. This is invaluable for the frontend as it provides a consistent error structure to work with.

2. **API Conventions:** It ensures your API behaves like a native Umbraco Management API, which is helpful for any developer familiar with Umbraco's backend.

**B. Revised Data Controller Skeleton (`BlogPostApiController.cs`)**

This more detailed skeleton shows how to inject necessary services and how to handle success and failure cases in a way that leverages the base class.

```csharp
// PRD: Backend Code - Detailed Data Controller
using Microsoft.AspNetCore.Mvc;
using Umbraco.Cms.Api.Management.Controllers;
using Umbraco.Cms.Core;
using Umbraco.Cms.Core.Models;
using Umbraco.Cms.Core.Services;
using Umbraco.Cms.Web.Common.Authorization;
using Umbraco.Extensions; // Needed for .HasValue()

[ApiController]
[Route("api/articulate/blogpost")]
public class BlogPostApiController : ManagementApiControllerBase
{
    private readonly IContentService _contentService;
    private readonly IUmbracoContextAccessor _umbracoContextAccessor;

    // NOTE TO DEV: Inject all services you need for content/media creation.
    public BlogPostApiController(
        IContentService contentService,
        IUmbracoContextAccessor umbracoContextAccessor)
    {
        _contentService = contentService;
        _umbracoContextAccessor = umbracoContextAccessor;
    }

    [HttpPost("create")]
    [Authorize(Policy = AuthorizationPolicies.ContentPermissionByResource)]
    public async Task<IActionResult> Create([FromBody] CreatePostRequest model)
    {
        // --- Step 1: Validate the input ---
        if (!_umbracoContextAccessor.TryGetUmbracoContext(out var umbracoContext) || umbracoContext.Content is null)
        {
            // This would be a server error. ManagementApiControllerBase will format it.
            return Problem("Could not get Umbraco Content Cache.", statusCode: 500);
        }

        var parentNode = umbracoContext.Content.GetById(model.ParentNodeId);
        if (parentNode is null)
        {
            // NOTE TO DEV: This is how you return a standard validation error.
            // The frontend will receive a detailed JSON object.
            return ValidationProblem("The specified parent node could not be found.");
        }

        // --- Step 2: Create the Content Item ---
        // Using "ArticulateMarkdown" as an example alias. Use the correct one.
        Attempt<IContent> createAttempt = _contentService.Create(model.Title, parentNode.Id, "ArticulateMarkdown");

        if (!createAttempt.Success)
        {
            return Problem("Failed to create the content item.", statusCode: 500);
        }

        var newPost = createAttempt.Result;

        // --- Step 3: Map Properties ---
        newPost.SetValue("articulate_markdown", model.Body); // Use correct property alias
        newPost.SetValue("tags", model.Tags); // Use correct property alias
        // ... set other properties ...

        // --- Step 4: Save and Publish ---
        var publishResult = _contentService.SaveAndPublish(newPost);
        if (!publishResult.Success)
        {
            return Problem($"Failed to publish the new post. Status: {publishResult.Status}", statusCode: 500);
        }

        // --- Step 5: Return Success ---
        // Get the new URL from the published content.
        var newUrl = newPost.Url();
        return Ok(new { success = true, url = newUrl.HasValue() ? newUrl : "#" });
    }

    // ... other methods like UploadMedia ...
}
```

    ```javascript
    // In the petite-vue app object:
    async handleFileSelect(event) {
        const file = event.target.files[0];
        if (!file) return;

        // 1. Insert a placeholder into the editor immediately for good UX.
        const placeholder = `![Uploading ${file.name}...]()`;
        this.easyMDE.codemirror.replaceSelection(placeholder);

        // 2. Prepare the data for the API.
        const formData = new FormData();
        formData.append('file', file);
        // NOTE: You might need to specify a media folder ID.
        // This could be a hardcoded ID or passed from the model.
        formData.append('parentFolderId', -1); // Example: root media folder

        try {
            // 3. Call the API to upload the file.
            // This assumes a 'uploadMedia' method exists on the BlogPostService.
            const result = await BlogPostService.uploadMedia(formData);
            const finalMarkdown = `![${file.name}](${result.mediaUrl})`;

            // 4. Replace the placeholder with the final Markdown.
            const currentText = this.easyMDE.value();
            this.easyMDE.value(currentText.replace(placeholder, finalMarkdown));

        } catch (error) {
            console.error('Upload failed:', error);
            // On failure, remove the placeholder.
            const currentText = this.easyMDE.value();
            this.easyMDE.value(currentText.replace(placeholder, ''));
            alert('Image upload failed.');
        }

        // Reset the input so the user can select the same file again.
        event.target.value = '';
    }
    ```
